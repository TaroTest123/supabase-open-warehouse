# データソース仕様 — TEPCO でんき予報

## 概要

東京電力パワーグリッド株式会社が公開する「でんき予報」の CSV データを使用する。
日々の電力需要実績と予測値が 30 分間隔で提供される。

## CSV URL パターン

| データ種別 | URL | 更新頻度 |
|-----------|-----|---------|
| 当日実績 | `https://www.tepco.co.jp/forecast/html/images/juyo-d-j.csv` | 5分ごと |
| 翌日予測 | `https://www.tepco.co.jp/forecast/html/images/juyo-d1-j.csv` | 1日1回 |
| 過去実績 | `https://www.tepco.co.jp/forecast/html/images/juyo-{YYYY}.csv` | 年次 |

> **注意**: URL パターンは変更される可能性がある。取込失敗時はアラートを出す設計にすること。

## CSV フォーマット

### エンコーディング

- **Shift_JIS (CP932)** — UTF-8 ではないので変換が必要
- Node.js では `TextDecoder('shift_jis')` または `iconv-lite` を使用
- Edge Function (Deno) では `TextDecoder('shift_jis')` が利用可能

### ヘッダ行の不規則性

TEPCO の CSV はヘッダ行が不規則で、標準的な CSV パーサーでは正しく読めないことがある。

```
// 実際の CSV 構造（例）
2024/7/24 の電力需要実績
DATE,TIME,実績(万kW),予測(万kW),使用率(%)
2024/7/24,0:00,2890,2850,62
2024/7/24,0:30,2780,2750,60
...
```

**対処法:**
1. 先頭のメタデータ行（タイトル行）をスキップする
2. ヘッダ行を検出してカラム名をマッピングする
3. データ行のみをパースする

### カラム定義

| カラム | 型 | 説明 | 例 |
|-------|---|------|-----|
| DATE | TEXT | 日付（YYYY/M/D 形式） | `2024/7/24` |
| TIME | TEXT | 時刻（H:MM 形式） | `14:00` |
| 実績(万kW) | TEXT | 電力需要実績（万kW単位） | `4567` |
| 予測(万kW) | TEXT | 電力需要予測（万kW単位） | `4500` |
| 使用率(%) | TEXT | 供給力に対する使用率 | `89` |

> カラム名は CSV によって微妙に異なる場合があるので、位置ベースでもパースできるようにする。

## データ処理上の注意点

### 単位変換

- CSV の値は **万kW** 単位
- DB には **MW (メガワット)** で格納する
- 変換式: `MW = 万kW × 10`
  - 例: `4567 万kW = 45,670 MW`

### 重複データの dedup 戦略

同じ日時のデータが複数回取り込まれる可能性がある（再取得時など）。

**戦略:**
1. **raw レイヤー**: UNIQUE 制約 `(date_str, time_str)` で重複 INSERT を防止。`ON CONFLICT DO UPDATE` で最新値に更新
2. **staging レイヤー**: dbt モデルで `ROW_NUMBER()` による dedup。`loaded_at DESC` で最新レコードを採用

### タイムゾーン

- TEPCO データは **JST (Asia/Tokyo, UTC+9)** 基準
- DB 格納時は `TIMESTAMPTZ` を使用し、JST を明示的に指定
- 表示時は JST で表示（ユーザーは日本のデータを扱っているため）

### 欠損データ

- 一部の時間帯でデータが欠損している場合がある
- 欠損は NULL として扱い、集計時は除外する
- チャットでの回答時に欠損がある旨をユーザーに伝える

## データ量の見積もり

| 期間 | レコード数 | 備考 |
|------|-----------|------|
| 1日 | 48 | 30分間隔 × 24時間 |
| 1年 | ~17,520 | 365日 × 48 |
| 10年 | ~175,200 | Phase 1-2 では十分小さい |

PostgreSQL のみで十分に対応可能なデータ量。インデックスを適切に設定すれば、Phase 1-2 で性能問題は発生しない。
